# Guide de workflow Git : Du chaos à la clarté

La plupart des équipes n'ont pas un problème Git. Elles ont un problème de _communication_ — et Git est l'endroit où ça se voit.

Un dépôt sans workflow convenu devient rapidement un cimetière de branches `fix-final-FINAL-v2`, de commits de merge couvrant trois jours de changements sans rapport, et d'une branche `main` qui casse tous les deux vendredis. Ça vous parle ?

La bonne nouvelle, c'est que la solution n'est pas compliquée. Une poignée de conventions convenues — écrites, légèrement appliquées, améliorées au fil du temps — transforme Git d'une source d'anxiété en un véritable superpouvoir pour votre équipe.

Ce guide couvre tout : les stratégies de branches, les conventions de commits, la culture des pull requests, les stratégies de merge, la résolution de conflits et l'automatisation. Adoptez ce qui convient à votre contexte, ignorez ce qui ne convient pas, et itérez à mesure que votre équipe grandit.

---

## Pourquoi le workflow compte plus que la maîtrise de l'outil

Vous pouvez connaître chaque commande Git par cœur et quand même livrer le chaos si votre équipe n'est pas d'accord sur les bases. Git est flexible par conception — il ne vous impose pas de workflow, ce qui signifie que le workflow est entièrement à vous de définir.

Voici ce qu'un bon workflow vous apporte vraiment :

**Un historique lisible.** Quand chaque commit est atomique et bien décrit, `git log` devient de la documentation. Vous pouvez retracer _pourquoi_ une décision a été prise, pas seulement _ce qui_ a changé. C'est inestimable six mois plus tard quand quelque chose casse et que personne ne se souvient du contexte original.

**Des revues ciblées.** Les petites pull requests ciblées sont examinées en profondeur. Les PR géantes avec 47 fichiers modifiés sont validées à l'aveugle parce que personne n'a la bande passante pour les lire vraiment. Un workflow qui impose des branches petites améliore directement la qualité du code.

**Des releases sans stress.** Quand `main` est toujours déployable et que chaque changement est conditionné à un pipeline CI au vert, les releases cessent d'être des événements et deviennent des non-événements. L'objectif est de rendre les déploiements _ennuyeux_.

**Un onboarding plus rapide.** Un nouveau développeur rejoignant une équipe avec un workflow documenté peut être productif en quelques jours. Une équipe sans conventions prend des semaines à naviguer avant que le nouveau fasse confiance à ses propres pushs.

## Choisir une stratégie de branches

Il existe plusieurs modèles de branches bien établis. Le bon dépend de la taille de votre équipe, de votre cadence de release et de votre modèle de déploiement. Voici les trois plus courants :

### Trunk-Based Development (recommandé pour la plupart des équipes)

Tout le monde travaille dans des branches de fonctionnalité à courte durée de vie (ou même directement sur `main` pour de petits changements) et intègre fréquemment dans `main` — idéalement au moins une fois par jour.

```
main
 ├── feat/user-auth          (vit 1–2 jours)
 ├── fix/login-redirect      (vit quelques heures)
 └── chore/update-deps       (vit 1 jour)
```

**Idéal pour :** Les équipes avec des pipelines CI/CD, des feature flags et une culture de changements petits et incrémentaux.

**Pourquoi ça fonctionne :** L'intégration fréquente signifie que les conflits de merge sont petits et détectés tôt. `main` est toujours prêt à être livré. Les déploiements sont à faible risque.

### GitHub Flow

Un modèle simplifié : créer une branche depuis `main`, ouvrir une PR, merger dans `main` une fois approuvée et verte. Simple, efficace, et ce que la plupart des petites et moyennes équipes devraient utiliser au départ.

```
main ──────────────────────────────────────────► (toujours déployable)
         │                         ▲
         └── feat/payment-flow ────┘
              (PR ouverte tôt, revue, mergée)
```

**Idéal pour :** Les équipes qui livrent en continu dans un seul environnement de production.

### Gitflow

Un modèle plus structuré avec des branches `develop`, `release` et `hotfix` à longue durée de vie aux côtés de `main`. Plus de cérémonie, plus d'isolation.

```
main       ─────────────────────────────────────► (production)
develop    ──────────────────────────────────────► (intégration)
            │                        ▲
            └── feat/dashboard ──────┘
release/1.2 ────────────────────────►
hotfix/1.1.1 ──────────────────────►
```

**Idéal pour :** Les produits avec des cycles de release planifiés, plusieurs versions supportées ou des contrôles QA stricts entre les environnements.

**Attention :** Gitflow ajoute une surcharge significative. Beaucoup d'équipes l'adoptent en pensant qu'il apportera de la rigueur, puis passent plus de temps à gérer les branches qu'à livrer des fonctionnalités. Commencez avec GitHub Flow et migrez seulement si vous avez vraiment besoin de cette structure.

## Conventions de nommage des branches

Des noms de branches cohérents permettent de comprendre instantanément l'objet d'une branche, qui en est responsable et quel ticket elle concerne. Choisissez une convention et appliquez-la.

Un pattern largement adopté :

```bash
# Format : <type>/<description-courte>
# Types : feat, fix, chore, refactor, docs, test, hotfix

feat/user-authentication
fix/cart-total-rounding-error
chore/upgrade-node-18
refactor/extract-payment-service
docs/api-authentication-guide
test/add-checkout-integration-tests
hotfix/critical-session-expiry-bug
```

Si votre équipe utilise un gestionnaire de projet, préfixez avec l'ID du ticket :

```bash
feat/PROJ-142-user-authentication
fix/PROJ-89-cart-total-rounding-error
```

Cela permet de passer facilement d'une PR au ticket original et vice-versa — un petit détail qui économise un temps surprenant.

**Appliquez-le avec un hook Git.** Ajoutez un hook `commit-msg` ou `pre-push` via Husky pour valider les noms de branches avant qu'ils n'atteignent jamais le dépôt distant :

```bash title=".husky/pre-push"
#!/bin/sh
branch=$(git symbolic-ref --short HEAD)
pattern='^(feat|fix|chore|refactor|docs|test|hotfix)\/[a-z0-9-]+'

if ! echo "$branch" | grep -qE "$pattern"; then
  echo "❌ Le nom de branche '$branch' ne correspond pas au pattern requis."
  echo "   Attendu : feat/description-courte ou fix/description-issue"
  exit 1
fi
```

## Écrire des commits qui aident vraiment

Les commits sont l'unité atomique de l'historique de votre projet. Un message de commit bien rédigé est un cadeau pour votre futur vous-même et pour chaque développeur qui viendra après vous.

### La spécification Conventional Commits

[Conventional Commits](https://www.conventionalcommits.org) est une spécification largement adoptée qui donne aux messages de commit une structure cohérente. De nombreux outils (générateurs de changelog, bots de release sémantique) sont construits pour la parser.

```
<type>(<portée>): <description courte>

[corps optionnel]

[pied de page optionnel]
```

**Types :**

| Type       | Quand l'utiliser                                                           |
| ---------- | -------------------------------------------------------------------------- |
| `feat`     | Une nouvelle fonctionnalité visible par les utilisateurs                   |
| `fix`      | Une correction de bug                                                      |
| `refactor` | Changement de code qui ne corrige pas un bug ni n'ajoute de fonctionnalité |
| `chore`    | Tâches de maintenance (dépendances, config, CI)                            |
| `docs`     | Documentation uniquement                                                   |
| `test`     | Ajout ou correction de tests                                               |
| `perf`     | Amélioration des performances                                              |
| `ci`       | Changements de configuration CI/CD                                         |
| `revert`   | Annule un commit précédent                                                 |

**Exemples concrets :**

```bash
# ❌ Mauvais — vague, sans contexte
git commit -m "fix bug"
git commit -m "update stuff"
git commit -m "WIP"
git commit -m "changes"

# ✅ Bon — type, portée et changement clairement définis
git commit -m "fix(auth): résoudre la session non persistante après rechargement de page"
git commit -m "feat(cart): ajouter les contrôles d'incrémentation/décrémentation de quantité"
git commit -m "chore(deps): mettre à jour react de 18.2 à 18.3"
git commit -m "refactor(api): extraire le timeout de requête dans une config partagée"
git commit -m "docs(readme): ajouter les instructions de configuration du développement local"
```

### Le corps et le pied de page

Pour les changements non triviaux, ajoutez un corps expliquant _pourquoi_, pas _quoi_ (le diff montre déjà quoi) :

```
fix(payments): empêcher la double facturation lors d'une nouvelle tentative réseau

Auparavant, relancer une requête de paiement échouée pouvait créer des
doublons de facturation car l'ID d'intention de paiement n'était pas
préservé entre les tentatives. Ajout d'une clé d'idempotence dérivée de
l'ID de commande + horodatage pour que le prestataire de paiement déduplique
de son côté.

Fixes: PROJ-412
Reviewed-by: @coequipier
```

### Les 7 règles d'un bon message de commit

1. Séparer le sujet du corps par une ligne vide
2. Limiter la ligne de sujet à 72 caractères
3. Utiliser le mode impératif (« corriger », pas « corrigé » ou « corrige »)
4. Ne pas terminer la ligne de sujet par un point
5. Utiliser le corps pour expliquer _quoi_ et _pourquoi_, pas _comment_
6. Référencer les issues et PR dans le pied de page
7. Un seul changement logique par commit — séparer les changements sans rapport

## Garder les branches petites et à courte durée de vie

C'est la pratique la plus impactante que vous puissiez adopter. Les branches à longue durée de vie sont l'endroit où les conflits de merge se multiplient, où les revues s'enlisent, et où la confiance dans le code se détériore.

**Visez des branches qui ne durent pas plus de 2 à 3 jours.** Si une fonctionnalité est trop grande pour tenir dans cette fenêtre, décomposez-la :

```
# Au lieu d'une seule branche géante :
feat/complete-checkout-flow   (3 semaines, 47 fichiers modifiés)

# Décomposez en :
feat/checkout-cart-summary    (2 jours, 8 fichiers)
feat/checkout-address-form    (1 jour, 5 fichiers)
feat/checkout-payment-step    (2 jours, 9 fichiers)
feat/checkout-confirmation    (1 jour, 4 fichiers)
```

Utilisez des **feature flags** pour livrer du code qui n'est pas encore visible par les utilisateurs. Cela vous permet de merger des fonctionnalités incomplètes dans `main` sans les exposer aux utilisateurs — en gardant votre branche à courte durée de vie sans précipiter la fonctionnalité.

```typescript
// Protégé par un feature flag — sûr à merger dans main avant que la fonctionnalité soit prête
if (featureFlags.isEnabled('new-checkout-flow', user)) {
  return <NewCheckoutFlow />;
}
return <LegacyCheckout />;
```

## Pull Requests : L'art de la revue

Une pull request n'est pas simplement un mécanisme de livraison de code — c'est un outil de communication. La description, la taille et la culture de revue autour des PR comptent toutes.

### Rédiger une bonne description de PR

Chaque description de PR doit répondre à trois questions :

1. **Qu'est-ce qui a changé ?** Un bref résumé du changement technique.
2. **Pourquoi ?** La motivation — quel problème cela résout-il ?
3. **Comment le vérifier ?** Les étapes pour que le relecteur teste le changement.

Utilisez un template de PR (`.github/pull_request_template.md`) pour en faire le format par défaut :

```markdown title=".github/pull_request_template.md"
## Ce qui a changé

<!-- Brève description du changement technique -->

## Pourquoi

<!-- La motivation. Lien vers l'issue si applicable -->

Closes #

## Tests

<!-- Comment avez-vous testé cela ? Que doit vérifier le relecteur ? -->

- [ ] Testé en local
- [ ] Tests unitaires ajoutés/mis à jour
- [ ] Tests d'intégration ajoutés/mis à jour
- [ ] Cas limites testés : \_\_\_

## Captures d'écran (si applicable)

<!-- Avant/après si l'UI a changé -->

## Checklist

- [ ] Le code respecte les conventions du projet
- [ ] Aucun console.log ou code de débogage laissé
- [ ] Aucune dépendance inutile ajoutée
- [ ] Documentation mise à jour si nécessaire
```

### Faire une revue de code comme un humain

La revue de code est l'une des activités à plus fort levier dans une équipe logicielle. Bien faite, elle diffuse les connaissances, détecte les bugs et améliore la conception. Mal faite, c'est une barrière qui ralentit tout sans apporter de valeur.

**Pour les relecteurs :**

- Distinguez les problèmes bloquants des suggestions non bloquantes. Utilisez des préfixes : `[bloquant]`, `[nit]`, `[question]`, `[suggestion]`.
- Posez des questions plutôt que d'exiger — « Que pensez-vous d'extraire ceci dans un helper ? » passe mieux que « Extrayez ceci. »
- Approuvez les PR qui sont assez bonnes, pas parfaites. Le parfait est l'ennemi du livré.
- Faites la revue rapidement. Une PR qui reste trois jours sans revue est une taxe de changement de contexte pour l'auteur.

**Pour les auteurs :**

- Gardez le diff petit. Une PR de moins de 400 lignes est revue ; une PR de 1 200 lignes est approuvée avec une prière.
- Annotez votre propre PR avant de demander une revue — guidez les relecteurs à travers les décisions clés pour qu'ils passent du temps sur la logique, pas sur l'orientation.
- Répondez à chaque commentaire, même juste pour dire « Fait » ou « Bon point, conservé tel quel car X ».
- Ne prenez pas les retours personnellement. Le relecteur répond au code, pas à vous.

### Recommandations sur la taille des PR

```
Moins de 200 lignes   → Revue rapide et approfondie, le jour même
200–400 lignes        → Acceptable, peut prendre un jour
400–800 lignes        → Envisager de diviser si possible
Plus de 800 lignes    → Devrait presque certainement être divisée
```

Ce ne sont pas des règles strictes — une PR de 600 lignes entièrement composée de types générés est acceptable. Mais une PR de 600 lignes avec une logique métier complexe dans six fichiers différents ne l'est pas.

## Stratégies de merge : Merge Commit vs. Squash vs. Rebase

La façon dont vous intégrez les branches dans `main` détermine l'aspect de votre historique. Chaque stratégie a ses compromis :

### Merge Commit

```
main:    A──B──────────────M
              \            /
feat:          C──D──E────
```

Préserve tous les commits individuels et montre la structure complète de la branche. L'historique est fidèle mais peut être bruyant avec de nombreux petits commits.

**Idéal pour :** Les équipes qui veulent préserver un historique atomique complet et retracer exactement comment une fonctionnalité a été construite.

### Squash and Merge

```
main:    A──B──────────────S
              \            ↑
feat:          C──D──E    (fusionnés en un seul commit)
```

Combine tous les commits de la branche en un seul commit sur `main`. L'historique est propre et linéaire — chaque entrée représente une fonctionnalité ou une correction complète.

**Idéal pour :** Les équipes qui préfèrent un historique `main` propre. Fonctionne bien quand les commits individuels d'une branche sont des commits WIP désordonnés qui n'apportent pas de valeur au log.

### Rebase and Merge

```
main:    A──B──C'──D'──E'
              ↑
feat:          C──D──E  (rejoués au-dessus de main)
```

Rejoue les commits de la branche au-dessus de `main` sans commit de merge. Produit un historique parfaitement linéaire.

**Idéal pour :** Les équipes qui veulent les commits individuels préservés _et_ un historique linéaire. Nécessite des commits propres et significatifs.

**La recommandation pratique :** Utilisez le **squash merge** pour la plupart des branches de fonctionnalité (une PR = un commit sur `main`), et le **merge commit** pour les branches de release où vous souhaitez conserver l'historique complet. Évitez le rebase sur les branches publiques/partagées — réécrire l'historique partagé crée des problèmes pour tout le monde.

## Gérer les conflits de merge comme un pro

Les conflits de merge sont inévitables. L'objectif est de minimiser leur fréquence et de les résoudre avec confiance quand ils surviennent.

**Minimisez les conflits en :**

- Gardant les branches à courte durée de vie et régulièrement synchronisées avec `main`
- Communiquant avec vos coéquipiers quand vous travaillez dans la même zone de la base de code
- Préférant les fichiers petits et ciblés aux grands fichiers monolithiques

**Quand un conflit survient :**

```bash
# Étape 1 : Synchronisez votre branche avec le dernier main
git fetch origin
git rebase origin/main
# ou
git merge origin/main

# Étape 2 : Git s'arrête sur les fichiers en conflit
# Ouvrez chaque fichier en conflit — cherchez les marqueurs de conflit :
# <<<<<<< HEAD
#   vos changements
# =======
#   leurs changements
# >>>>>>> origin/main

# Étape 3 : Résolvez manuellement (ou utilisez un outil de merge)
git mergetool   # ouvre votre outil configuré (VS Code, IntelliJ, vimdiff, etc.)

# Étape 4 : Marquez comme résolu et continuez
git add <fichier-résolu>
git rebase --continue   # ou git merge --continue
```

**Configurez VS Code comme outil de merge :**

```bash
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'
```

**La règle d'or de la résolution de conflits :** en cas de doute, parlez à la personne dont vous mergez le code. Une conversation de 2 minutes vaut mieux que 30 minutes à deviner ce que son changement voulait dire.

## Automatisation : Laissez la machine faire les parties ennuyeuses

Un bon workflow est appliqué par l'automatisation, pas par la mémoire ou les bonnes intentions. Voici la chaîne d'outils qui vaut la peine d'être mise en place :

### Husky + lint-staged (hooks pre-commit)

Lancez le linting et le formatage sur les fichiers en staging avant chaque commit :

```bash
npm install --save-dev husky lint-staged
npx husky init
```

```json title="package.json"
{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,mdx,css}": ["prettier --write"]
  }
}
```

```bash title=".husky/pre-commit"
#!/bin/sh
npx lint-staged
```

### Commitlint (applique le format des messages de commit)

```bash
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

```javascript title="commitlint.config.js"
export default {
  extends: ["@commitlint/config-conventional"],
  rules: {
    "type-enum": [
      2,
      "always",
      [
        "feat",
        "fix",
        "chore",
        "refactor",
        "docs",
        "test",
        "perf",
        "ci",
        "revert",
      ],
    ],
    "subject-case": [2, "always", "lower-case"],
    "subject-max-length": [2, "always", 72],
  },
};
```

```bash title=".husky/commit-msg"
#!/bin/sh
npx --no -- commitlint --edit "$1"
```

### Pipeline CI (GitHub Actions)

Chaque push et PR doit lancer la suite de tests complète, le linting et la vérification de types. Rien ne merge dans `main` tant que la CI n'est pas au vert :

```yaml title=".github/workflows/ci.yml"
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - run: npm ci

      - name: Vérification des types
        run: npm run type-check

      - name: Linting
        run: npm run lint

      - name: Tests
        run: npm run test -- --coverage

      - name: Build
        run: npm run build
```

Exigez que la CI soit au vert et qu'au moins une revue soit approuvée avant de merger — appliquez cela via les règles de protection de branches dans les paramètres GitHub, GitLab ou Bitbucket.

## Commandes Git utiles à connaître

Au-delà des bases, ces commandes reviennent régulièrement et méritent d'être dans vos réflexes :

```bash
# Voir un graphe visuel de l'historique de vos branches
git log --oneline --graph --all

# Annuler le dernier commit en gardant les changements en staging
git reset --soft HEAD~1

# Mettre en stash avec une description
git stash push -m "WIP: refactoring auth à moitié fait"
git stash list
git stash pop

# Trouver le commit qui a introduit un bug (recherche binaire)
git bisect start
git bisect bad                  # le commit actuel est cassé
git bisect good v1.2.0          # ce tag fonctionnait
# Git checkout le point médian — testez-le, puis :
git bisect good   # ou git bisect bad
# Répétez jusqu'à ce que le commit coupable soit identifié
git bisect reset

# Cherry-pick un commit spécifique vers une autre branche
git cherry-pick <hash-du-commit>

# Voir qui a modifié en dernier chaque ligne d'un fichier
git blame src/auth/session.ts

# Chercher dans l'historique des commits une chaîne de caractères
git log -S "nomDeFonction" --source --all

# Nettoyer les branches locales qui n'existent plus sur le dépôt distant
git fetch --prune
git branch -vv | grep ': gone]' | awk '{print $1}' | xargs git branch -d
```

## Une checklist pratique pour l'équipe

Imprimez-la, mettez-la dans votre README ou déposez-la dans le wiki de votre équipe :

### ✅ Branches

- [ ] Créer une branche depuis `main` pour chaque changement — ne jamais travailler directement sur `main`
- [ ] Les noms de branches suivent la convention convenue (`feat/`, `fix/`, etc.)
- [ ] Les branches sont à courte durée de vie (viser moins de 3 jours)
- [ ] Synchronisation fréquente avec `main` pour éviter les grands conflits de merge

### ✅ Commits

- [ ] Les messages de commit suivent le format Conventional Commits
- [ ] Chaque commit représente un seul changement logique
- [ ] Aucun commit avec des messages comme « WIP », « fix », « test », « asdf »
- [ ] Les grands changements sont découpés en une série de commits significatifs

### ✅ Pull Requests

- [ ] La description de PR explique quoi, pourquoi et comment vérifier
- [ ] La PR est assez petite pour être revue en moins de 30 minutes
- [ ] L'issue ou le ticket est lié
- [ ] Les tests sont ajoutés ou mis à jour
- [ ] Des captures d'écran sont incluses pour les changements UI
- [ ] La CI est au vert avant de demander une revue

### ✅ Revues

- [ ] Revue effectuée dans les 24 heures suivant la demande
- [ ] Les commentaires distinguent les points bloquants des non-bloquants
- [ ] Chaque commentaire reçoit une réponse avant le merge
- [ ] Les approbations sont significatives, pas des tampons automatiques

### ✅ Automatisation

- [ ] Les hooks pre-commit Husky lancent le lint et le format
- [ ] Commitlint applique le format des messages de commit
- [ ] La CI lance la vérification de types, le lint, les tests et le build sur chaque PR
- [ ] La protection de branche exige la CI au vert + au moins 1 approbation

## Conclusion

L'objectif n'a jamais été « Git parfait ». L'objectif est moins de surprises, une collaboration plus rapide et une base de code en laquelle vous avez assez confiance pour déployer un vendredi après-midi.

Commencez par les bases : convenez d'une stratégie de branches, écrivez votre convention de commits, revoyez les PR rapidement et lancez la CI sur tout. Documentez votre workflow dans votre README pour que chaque nouveau développeur hérite de la culture dès le premier jour plutôt que de la découvrir après un mois.

Les workflows s'améliorent grâce à des rétrospectives honnêtes. Chaque conflit de merge douloureux, chaque PR approuvée à la va-vite qui a introduit une régression, chaque `git log` qui vous laisse plus confus qu'avant — ce sont des données. Apportez-les à votre équipe, ajustez les règles et continuez à itérer.

Un historique Git propre est une marque de considération pour tous ceux qui travaillent dans la base de code — y compris la version de vous-même qui déboguera un incident en production à 23h dans six mois.
